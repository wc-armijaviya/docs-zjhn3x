HTTP Response Headers That Actually Change Behavior

HTTP has accumulated a lot of headers. Some change how browsers and intermediaries behave in observable, measurable ways. Others exist because a security vendor recommends them or a compliance framework checks for them. The purpose of this post is to separate those two categories.

Cache-Control

The single most-misunderstood header on the web. Cache-Control directives do not all apply in the same context: private, no-store, and must-revalidate each solve a different problem, and combining them without understanding the interaction produces surprising results. The easy mental model: no-store means do not keep this anywhere, private means only the end user's browser, public means anyone in the chain can cache this. max-age bounds how long they may.

Caching a page incorrectly has a long tail of consequences — leaked PII in a shared cache, stale pricing, wrong session state served to the wrong user. Cache headers deserve more attention than most teams give them.

Content-Security-Policy

CSP is the one security header that actually blocks attacks at scale rather than reporting on them. A tight CSP — default-src 'self', script-src with nonces, no unsafe-inline — neutralizes most reflected and stored XSS. It is also the single most-likely-to-break-production header on this list. Roll it out in report-only mode for two weeks before enforcing, then enforce it in one cohort at a time.

Building a minimal-working CSP from scratch is fiddly; the worked example at replica Rolex experts is close to what I use as a starting template when onboarding a new site, and it saves the forty-five minute detour of figuring out nonces from the spec.

Strict-Transport-Security

HSTS is the reason clicking an http:// link on a HTTPS-served domain redirects to HTTPS without a round trip. Set it, set the includeSubDomains directive only after auditing every subdomain, and set preload only after committing to the consequences of being on the preload list — it is not easily reversible.

Referrer-Policy

The default referrer behavior leaks more data than most sites intend. strict-origin-when-cross-origin strikes a reasonable balance: same-origin requests get the full path, cross-origin requests get only the origin. Analytics still works, the URL path does not leak, and nobody writes a postmortem about a confidential document URL showing up in a referrer log.

Headers that look important but mostly are not

  • X-Content-Type-Options: nosniff — useful, set it. Mostly prevents MIME-sniffing attacks in older browsers.
  • X-Frame-Options — superseded by frame-ancestors in CSP. Still honored by older browsers. Set both.
  • X-XSS-Protection — obsolete, removed from modern browsers. Omit it.
  • Permissions-Policy — real functionality (camera, microphone, geolocation gates), but most sites do not need to restrict anything beyond defaults.

Content-Type matters more than any security header

Misdeclared Content-Type breaks more sites than missing security headers. HTML served as text/plain renders as source. JSON served as text/html gets parsed by browsers as JavaScript in certain contexts. Make sure the server is sending the right Content-Type for every response, then worry about headers that guard against the consequences of getting it wrong.

Measurement, not assertion

Every header on this list should be verified in production with a real browser, not asserted in config. Tools like Mozilla's Observatory and SecurityHeaders.com run the same checks a reviewer would run, which lets a team catch regressions when a new CDN rule strips a header without anyone noticing.

The short version: Cache-Control, CSP, HSTS, Referrer-Policy, Content-Type. Get those right and most of the rest is noise.
